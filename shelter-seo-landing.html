02
Executive summary

Ключовий висновок

Новий сайт притулку потрібно запускати як SEO- та data-ready платформу: із чистою архітектурою URL, вимірюванням ключових дій від першого дня, мобільною першістю, керованим CMS і міграцією через повну redirect map. Google прямо рекомендує логічну структуру сайту, зрозумілі URL, crawlable internal links, коректні канонікали, sitemap, а при зміні URL — підготовлений URL mapping і 301 redirects.

Відповідь на три питання клієнта

Measure · Manage · Mobile

По-перше, ефективність і usability потрібно міряти через стек GA4 + Search Console + GTM + Hotjar або FullStory + Core Web Vitals + Lighthouse + Looker Studio + A/B testing tool; Google окремо рекомендує дивитися GA4 і Search Console разом, а Core Web Vitals — через Search Console та web.dev-інструменти.

По-друге, якщо головна задача — швидко редагувати контент, сторінки тварин, події, FAQ, локації та SEO-поля без зайвого dev-овергеду, найкращий баланс дає WordPress + Gutenberg + ACF; Webflow підходить для швидкого маркетингового сайту, Strapi — коли потрібен headless і сильна dev-команда, Shopify — лише якщо e-commerce стає ядром проєкту.

По-третє, адаптивність має бути не "додатком після дизайну", а базовою вимогою: mobile-first, однаковий основний контент і metadata на mobile/desktop, responsive images через srcset, коректний viewport, перевірка Core Web Vitals та QA на реальних пристроях.

Що рекомендувати на старті
WordPress (Gutenberg + ACF) + GA4 + GTM + GSC + Hotjar + Looker Studio + Lighthouse QA + JSON-LD schema + CDN/HTTPS/cache + повна redirect map. Це дає найкраще співвідношення керованості контентом, SEO-контролю, вартості підтримки та швидкості запуску для неприбуткового або service-oriented сайту притулку.

CMS

CMS та адмін-панель

4.1 — Що CMS має вміти обов'язково

Для притулку CMS має підтримувати не лише "сторінки", а структурований контент: pet profiles, event pages, donation pages, location pages, FAQs, volunteer roles, documents. Мінімальні вимоги: ролі й права доступу, чернетки, revision history, SEO title/meta, open graph, canonical, redirects, sitemap, schema fields, медіабібліотека, content relationships, page scheduling, form integration і можливість масштабувати шаблони без ручного дублювання. Також потрібен чіткий workflow: draft → SEO/content review → publish.

4.2 — Рекомендація
Рекомендую
WordPress + Gutenberg + ACF

Чому: для притулку критичні не лише красиві сторінки, а й редакторська незалежність, можливість швидко додавати/редагувати structured pages, хороші ролі доступу, контрольовані SEO-поля, масштабування шаблонів і помірна вартість підтримки. WordPress уже має ролі, block editor і базовий sitemap у core; ACF додає чисту модель custom fields. Це робить WordPress найпрактичнішим вибором, якщо CMS ще не визначено.

Адаптивність

Адаптивність та responsiveness

5.1 — Що означає "site is truly responsive"

Responsive design — це не просто "сайт відкривається на телефоні". Для Google важливо, що індексація й ранжування йдуть за mobile version, а значить головний контент, metadata і structured data на mobile мають бути еквівалентні desktop. Можна змінювати layout, ховати блоки в акордеони й таби, але не зменшувати змістовну цінність мобільної версії. Якщо mobile має менше первинного контенту, Google прямо попереджає про можливу втрату трафіку.

viewport потрібен, щоб responsive CSS і media queries працювали коректно на вузьких екранах. srcset дозволяє віддавати різні версії зображень залежно від екрана і мережі. WCAG 2.2 вимагає мінімальний target size 24×24 CSS px. Device Mode у DevTools корисний для швидкої симуляції, але Chrome прямо попереджає: це лише наближення, фінальний QA треба робити на реальних мобільних пристроях.

5.4 — Що найчастіше ламає adaptive + SEO
  • mobile-версія без частини основного контенту;
  • інший title / meta description на mobile;
  • lazy-load primary content only on interaction;
  • важкі hero images/video без responsive delivery;
  • sticky bars/popups, які зміщують layout і б'ють по CLS;
  • фільтри / меню / accordions, до яких не можуть дістатися користувачі або Google.
5.5 — QA protocol
  • Пройти кожний шаблон у Chrome DevTools Device Mode.
  • Перевірити Lighthouse по mobile для homepage, adopt list, pet profile, donate, volunteer, event template.
  • Перевірити реальні пристрої: small phone, large phone, tablet.
  • Перевірити form UX: поле за полем, помилки, scroll to error, submit visibility.
  • Перевірити CWV у field data після запуску, а не лише в lab.
06
Технічне SEO

Технічне SEO та обов'язкові імплементації

6.1 — URL architecture

Google рекомендує просту і зрозумілу URL-структуру. Для shelter-сайту базова схема може бути такою:

/ /adopt/ /adopt/dogs/ /adopt/cats/ /pets/dog/{pet-name-id}/ /pets/cat/{pet-name-id}/ /donate/ /volunteer/ /foster/ /events/ /events/{event-slug}/ /services/spay-neuter/ /lost-found/ /locations/{city}/ /faq/ /about/ /contact/

Логіка: один template = один intent. Це спрощує indexation, canonicalization, titles, internal links і аналітику по шаблонах. Google рекомендує логічну організацію сайту, прості URL і crawlable links через <a href>.

6.2 — Technical SEO mandatory list
  • Canonical на всіх indexable сторінках; внутрішні посилання вести на canonical URL.
  • Titles / meta descriptions: унікальні, page-specific; Google може переписувати title/snippet, якщо сторінка подає слабкі або суперечливі сигнали.
  • H1 / headings: один чіткий головний заголовок, послідовна ієрархія секцій. Google радить виносити слова, які люди реально шукають, у title, main heading, alt text та link text.
  • Internal linking: важливі сторінки мають бути reachable навігаційно, без JS-only links.
  • Robots / noindex: robots.txt — для керування crawl traffic, а не для приховування сторінок із Search; для виключення з індексу потрібен noindex або захист доступу.
  • Sitemap: включати лише canonical URLs; подавати через Search Console; тримати актуальним.
  • Hreflang: лише якщо реально є multilingual / multi-regional site; для кожної мовної версії — окремий URL. Google не використовує hreflang або HTML lang для визначення мови сторінки, але hreflang допомагає пов'язати локалізовані версії.
  • Structured data: JSON-LD, валідовано через Rich Results Test / Schema Markup Validator.
  • HTTPS everywhere + без mixed content.
  • CDN + HTTP caching для статики й media.
6.3 — Structured data plan

AnimalShelter існує в Schema.org як subtype LocalBusiness. Google офіційно підтримує Organization, LocalBusiness, Event та інші типи в Search Gallery. Для FAQ є важливе обмеження: rich results доступні лише для well-known government/health sites, тому для чотириногого притулку FAQ markup варто використовувати радше як semantic markup, а не як гарантований SERP gain. Окремого Google-rich-result типу для DonateAction Google не документує в supported Search Gallery, тому donation schema слід розглядати як semantic layer, а не як джерело окремого rich result — це аналітичний висновок на основі Google Search Gallery та Schema.org vocabulary.

6.4 — On-page SEO checklist
  • Унікальний title для кожної сторінки.
  • Унікальна meta description для комерційних та key content pages.
  • Один чіткий H1.
  • Логічні H2/H3 під контентні секції.
  • Ключові слова в title, H1, alt text, anchor text — природно, без спаму.
  • Above-the-fold ясний CTA: Adopt / Donate / Volunteer / Foster / Contact.
  • Related content blocks: similar pets, upcoming events, donation CTA, volunteer CTA.
  • Breadcrumbs на глибоких сторінках.
  • Indexable pages мають бути linked з навігації або релевантних hub pages.
6.5 — Redirect strategy

Google для site moves рекомендує готувати URL mapping до релізу, а не після нього. Правильна міграція включає: old-to-new mapping, 301 redirects, оновлення sitemap, canonical, internal links і post-launch validation.

6.6 — Crawl budget — чи потрібно хвилюватися

Для більшості shelter-сайтів crawl budget не є першочерговою проблемою. Google прямо пише: якщо сайт не дуже великий і сторінки зазвичай скануються в день публікації, вистачає актуального sitemap та регулярної перевірки indexing coverage. Crawl-budget optimization стає важливою лише тоді, коли з'являються тисячі pet URLs, faceted filters, параметричні дублі або дуже часті оновлення каталогу.

6.7 — Local SEO

Притулок має керувати присутністю через Google Business Profile: адреса, години, телефон, сайт, фото, категорія, reviews, verification. Google вказує, що verified Business Profile допомагає показувати правильну інформацію в Search і Maps, а Knowledge Panel / business details також підтримуються через site ownership у Search Console та structured data.

6.8 — Security / hosting
  • HTTPS за замовчуванням і повна відмова від mixed content.
  • CDN перед origin для статики й media — зменшує latency, серверне навантаження і покращує TTFB/LCP.
  • Нормальний Cache-Control / HTTP cache headers.
  • Автооновлення ядра/плагінів/тем, якщо CMS це дозволяє; у WordPress це підтримується з core-рівня.

  • Не враховано конкретні інтеграції shelter software / CRM / donation processor; якщо вони є, це може змістити вибір у бік headless або спеціальних модулів.

Summary

На запуске обязательны: индексируемые посадочные, карточки животных, Google Search Console, GA4, XML sitemap, простые URL, <a href>-ссылки, mobile-first parity контента и метаданных.
